## Sixty seconds, honestly

```bash
npm run new-pack -- space-cats "Space Cats" 🐈   # a complete, valid starter game
npm run build && node tools/validate-packs.mjs space-cats
npx http-server dist                              # → play at /space-cats/
```

That's a registered, playable, gate-passing game — yours to reshape. The
engine keeps everything an LLM (or a tired human) gets wrong across
iterations: a stable seeded game loop, saves and resume, validation with
repairable errors, balance simulation, and a phone-first UI shell. The pack —
the world, the words, the numbers — is the part you write.

Two shipped games run on it today, unchanged engine:
**[Big Break](https://sandstreampop.github.io/big-break/)** (a satirical music
career) and **[The Villa](https://sandstreampop.github.io/big-break/love-island/)**
(a reality-TV season). Both are reference implementations you can read.

## What you're building on

The core runs one lifecycle — deck assembly → draw a card → swipe left or
right → roll to a `bad` / `good` / `incredible` tier → apply effects → segment
breaks → crossroads → finale — over a seeded, resume-safe RNG, across a
manifest-declared list of run segments (three acts, six weeks — the pack's
call). It knows nothing about *your* game.

A game is a **`Pack`**: a manifest, a deck, optional subsystems, and a
presenter. The whole surface is typed, and a truly unknown key is a compile
error — so the tier of a swipe outcome is exactly one of three things, and
nothing else:

```ts twoslash
type Tier = 'bad' | 'good' | 'incredible';
const rolled: Tier = 'good';
//    ^?
```
