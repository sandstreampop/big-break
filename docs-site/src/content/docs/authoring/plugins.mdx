A [`Plugin`](/big-break/docs/reference/types/interfaces/plugin/) is how a genre
adds its **own state and lifecycle** ŌĆö a chart, a reputation track, an inventory ŌĆö
without the engine knowing about it. The engine dispatches to your plugin at
fixed lifecycle points; your plugin does whatever your genre needs there.

## The smallest possible subsystem

A subsystem can be tiny. Here is the whole idea in a dozen lines: a plugin that
owns one counter, applies one effect verb, and narrates at act breaks.

```ts twoslash
interface Plugin {
  id: string;
  effectVerbs?: string[];
  onEffect?(state: any, effects: any, ctx: any): void;
  onActBreak?(state: any, act: number, notes: string[]): void;
}
// ---cut---
const favorPlugin: Plugin = {
  id: 'favor',
  effectVerbs: ['favor'],

  onEffect(state, effects, ctx) {
    const gained = effects.favor || 0;
    if (gained) {
      state.favor = (state.favor || 0) + gained;
      // record the delta so the trace/UI shows it (same shape as resources)
      ctx.deltas?.push({ key: 'favor', amount: gained });
    }
  },

  onActBreak(state, _act, notes) {
    const n = state.favor || 0;
    if (n > 0) notes.push(`You've banked ${n} favor.`);
  },
};
```

That's a complete subsystem. It declares the verb it owns (`effectVerbs:
['favor']`), maintains `state.favor` in `onEffect`, records a delta so the UI can
show it, and adds a note at each act break. A summit can then gate on `favor` ŌĆö
the manifest reads it through `gateValue` with no engine change.

## The hook lifecycle

Every hook is optional. Implement only the ones your subsystem needs. In run
order:

| Hook | When | Typical use |
|---|---|---|
| `onConstruct(state, rng)` | run start, before perks | seeded setup draws (weather, rival) |
| `onRunStart(state, rng)` | end of `newRun` | starting inventory (the notebook demo) |
| `modifyEffects(state, effects, ctx)` | before an outcome lands | adjust the payload (home-venue bonus) |
| `applyResource(res, effects, state, ctx)` | resource apply step | own a manifest resource's arithmetic |
| `onEffect(state, effects, ctx)` | after resources | handle your own effect verbs |
| `afterResolve(state, result, ctx)` | after a card resolves | react to the whole resolution |
| `onActBreak(state, act, notes)` | each act break | periodic ticks, income, audits |
| `onTick(state, notes)` | a periodic tick | chart weeks and the like |
| `onFinale(state)` | as the finale is judged | one last tick before scoring |

<Aside type="caution" title="Ordering is deterministic ŌĆö and load-bearing">
Plugins fire in ascending `priority` (default 0), ties broken by registration
order. Because subsystems can consume RNG, their order fixes the seeded draw
stream ŌĆö so the golden masters pin it. The music pack registers `[band, songs]`
specifically so band quirks draw before the chart tick. Set `priority`
deliberately, and don't reorder casually.
</Aside>

## The per-resolution context

Hooks that run during a card resolution receive a
[`PluginContext`](/big-break/docs/reference/types/interfaces/plugincontext/) ŌĆö the
resolution's shared scratch space:

- `ev`, `choice`, `tier` ŌĆö what's being resolved;
- `rng` ŌĆö the seeded generator (use this, never `Math.random`, or you break
  replay);
- `deltas` ŌĆö push structured deltas here so the UI can stage them;
- per-card scratch fields a subsystem stashes for itself across the hooks of one
  card.

## Owning a resource

If a resource needs bespoke arithmetic, a plugin can *own* it with
`applyResource`: return the delta to record, `0` to claim it with no delta, or
`undefined` to decline (the engine tries the next plugin, then a generic add).

## What a plugin OWNS (the contract, in full)

Beyond the lifecycle hooks, a plugin makes three kinds of **declarations** ŌĆö
each one is checked by the pack validator and the cross-pack invariants, so an
undeclared use fails the build instead of silently no-opping:

- **`effectVerbs`** ŌĆö the effect keys this plugin handles in `onEffect`. A
  card may only name a manifest stat/resource, a core verb (`burnout`,
  `addFlag`, `removeFlag`, `chainEventId`, `addPromise`), or a verb exactly
  one plugin declares here.
- **`requires`** ŌĆö the deck-eligibility predicates this plugin registers:
  `{ venueIs: (state, arg) => boolean }` lets a card gate on
  `requires: { venueIs: 'basement' }`. Each key has exactly **one** owner
  across the pack (two plugins claiming the same key is a validation error).
  Predicates must be pure and consume no RNG ŌĆö the deck evaluates them freely.
- **`stateDefaults`** ŌĆö the run-state slots this plugin owns, with fresh-run
  defaults. `newRun` applies them generically (arrays copied per run), and
  keys materialize top-level on `state`, so `gateValue` resolves them ŌĆö a
  `winGates`/`failStates`/`requires` gate can read a plugin-owned counter with
  no engine change.

## The modify-hooks (bending a core mechanic)

Where the lifecycle hooks *react*, the modify-hooks let a subsystem *bend* a
mechanic the engine keeps. The engine folds each in priority order; returning
`undefined` (or omitting the hook) contributes the fold's identity, so an
unimplemented hook costs nothing:

| Hook | Fold | Bends |
|---|---|---|
| `modifyRoll(state, choice, ctx)` | sum | additive roll bonus (push items whose bonus fired onto `ctx.applied`) |
| `modifyJitter(state, [lo,hi], ctx)` | chain | the roll's jitter band |
| `modifyBurnout(state, v, ctx)` | chain | a card's burnout delta |
| `gainHooks(state)` | collect | per-resolution multiplier bags (`statGainMult`, `burnoutGainMult`, `burnoutHealMult`) |
| `weightDeck(state, ev, w)` | chain | a card's draw weight |
| `refineDeck(state, pool, ctx)` | chain | force a scheduled category into the pool (the shop-slot mechanism) |
| `modifyActLength(state, act, n)` | chain | how many cards a segment runs |
| `scoreMult(state)` | product | a Legacy Points multiplier |
| `blocksEncore(state)` | OR | disable the Encore mechanic this run |

`modifyRoll` is also called for the **risk tell** (the odds preview), so it
must be pure and consume no RNG.

## The pattern to copy

`favor` above and `songs` are the same shape at two sizes. To add a subsystem:

1. Declare the verbs it owns in `effectVerbs`.
2. Keep its state on `state` (a namespaced key or two).
3. Handle its verbs in `onEffect`; own a resource with `applyResource` if needed.
4. Do periodic work in `onActBreak` / `onTick` / `onFinale`, pushing player-facing
   `notes` and structured `deltas`.
5. Register it in your pack's `plugins` array, with a deliberate `priority`.

The engine never learns your subsystem exists ŌĆö it just calls the hooks.
