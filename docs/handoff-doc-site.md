# Handoff — build the engine's documentation site

*For the next session. Goal: stand up a state-of-the-art documentation site for
this engine, aimed at **game developers who will build their own games on it**
(not maintainers of this repo). Autogenerated from the types/code where
possible, with AI-written prose as the connective tissue. This doc gives you
the mental model and a concrete plan; explore the codebase yourself for the
details — it's small and the source comments are unusually thorough.*

---

## 0. What you're documenting, in one breath

Big Break is a **3-act swipe-roguelike engine**. The core (`js/engine.ts`) is
genre-neutral: it runs the lifecycle (deck assembly → draw → swipe L/R → roll
to a bad/good/incredible tier → apply effects → act breaks → crossroads →
finale), a seeded resume-safe RNG, and nothing else. A *game* is a **`Pack`** —
a manifest + a deck + optional plugins + a presenter. Two games already ship on
it (a music-career game and a reality-show murder mystery) and a deliberately
minimal **probe pack** proves the core needs no genre. The headline promise for
your audience:

> **Adding a genre edits new files only — no shared type, no engine line.**

That sentence is the spine of the whole doc site. Everything else is in service
of showing a dev how to do exactly that.

---

## 1. The API surface (your single source of truth)

Almost the entire public contract lives in **`js/types.ts`**, and it is already
richly commented. This is what the autogenerated reference should be built from.
The types a pack author touches, roughly in the order they meet them:

- **`Pack`** — the composition root. Required: `id`, `manifest`, `events`,
  `tutorialEvents`, `instruments`, `instrumentById`. Everything else is an
  *optional capability* the engine feature-detects (`plugins`, `presenter`,
  `perks`, `interstitials`, `tutorialStart`, and the subsystem providers
  `accessories`/`arcs`/`contractById`/`rollWeather`/… ). The probe pack shows
  the true minimum; the music pack shows the maximum.
- **`PackManifest`** — the taxonomy that *is* the genre: `stats`, `resources`,
  `paths`, `winGates`, `statMeta`, `resourceMeta`.
- **`GameEvent` / `Choice` / `Outcome` / `Effect` / `Requires`** — the authored
  card schema (the deck).
- **`Effect`** — the payload vocabulary. Note the important design point for the
  docs: `Effect` is **open via declaration merging**. The core interface lists
  only genre-neutral keys; each pack adds its own stat/verb keys from its own
  file (`declare module '../types.js' { interface Effect { … } }`). This is the
  OCP win — document the pattern with the probe/music/mystery examples.
- **`Plugin` / `PluginContext`** — subsystems. Hooks: `onConstruct`,
  `onRunStart`, `modifyEffects`, `onEffect`, `afterResolve`, `onActBreak`,
  `onTick`, `onFinale`, plus `applyResource` (own a manifest resource) and
  `effectVerbs`/`priority` metadata.
- **`Presenter`** — the UI-facing flavor (endings, exit interviews, epilogue,
  headlines, DMs, trophies, wall). This is how a genre reaches its finale
  screen without music's copy.
- **`PerkDef`, `InterstitialRule`, `TutorialStart`, `PathDef`, `StatMeta`,
  `RunState`, `Song`** — the supporting cast.

The **engine's public functions** (`js/engine.ts` exports) are the runtime API a
*host* (UI/driver) calls, not something a pack author usually touches:
`newRun`, `useContentPack`, `drawNextCard`, `choiceOdds`, `resolveSwipe`,
`advance`, `commitPath`, `evaluateFinale`, `runSummary`, `legacyPoints`,
`gateValue`, `mulberry32`/`stateRng`. Document these as the "embedding the
engine" / "driving a run" reference, separate from the pack-authoring surface.

## 2. Worked examples that already exist (lean on these hard)

The best docs teach from real, runnable code. You have three packs at three
sizes — use them as the running examples throughout:

- **`js/packs/probe.ts`** — the minimal pack. One stat, one resource, one
  summit, a ~12-card generated deck, **zero optional capabilities**. This *is*
  the quickstart's end state. It's ~90 lines and reads top-to-bottom.
- **`js/packs/mystery.ts`** (+ `mystery-manifest.ts`, `mystery-presenter.ts`,
  `data/mystery-events.ts`, `plugins/clues.ts`) — a real second genre: a custom
  manifest, its own deck, one domain plugin (`clues`), a presenter, and only the
  capabilities it uses. This is the "realistic small game" example.
- **`js/packs/music.ts`** (+ `music-manifest.ts`, `music-presenter.ts`,
  `music-perks.ts`, `plugins/{venue,rival,band,songs}.ts`, `data/*`) — the
  full-featured game: every subsystem, perks, interstitials, tutorial. The
  "everything the engine can do" reference.

`js/packs/registry.ts` lists all three — the one place a genre is registered.

## 3. Recommended doc-site architecture

Two layers, stitched into one site:

1. **Autogenerated API reference** — point a tool at `js/types.ts` and the
   `js/engine.ts` public exports. **TypeDoc** is the obvious fit (the repo is
   already TS with a pinned toolchain). The type comments are dense and
   accurate, so the generated reference will be genuinely useful with little
   effort. Consider a TypeDoc theme that renders interfaces cleanly, and pull
   the doc-comments through verbatim.
   - Caveat to plan around: those comments were written for *maintainers*
     (they explain byte-green rationale, phase tags, RNG-order constraints). For
     the site you'll want the prose layer to **reframe for a game-dev audience**
     ("how do I add a stat?") rather than surface the maintenance history. Don't
     dump maintainer rationale on a newcomer.

2. **Handwritten prose (AI-authored) chapters** — a modern docs SSG
   (Docusaurus / VitePress / Starlight are all reasonable; VitePress/Starlight
   are lighter and pair nicely with a TS project). Prose links *into* the
   generated reference rather than restating it. Keep code samples as real,
   compilable excerpts from the three packs so they never drift.

The gold standard to emulate: the quickstart gets someone to a *playable* pack
fast, then chapters go deep one subsystem at a time, each opening with "what
problem this solves" before the API. Think Stripe/Tailwind/Astro docs quality.

## 4. Proposed information architecture

A first-draft table of contents (reorder as you learn the codebase):

1. **Quickstart — your first pack in ~10 minutes.** Copy the probe pack, rename
   a stat, add a card, register it, run it. End state: a playable one-summit
   game. (Use `tools/pack-core.mjs`'s generic driver or the browser to run it.)
2. **Core concepts.** The run lifecycle (3 acts, crossroads, swipe L/R,
   bad/good/incredible tiers, encore, pity, hot streaks, flashpoints), the
   seeded RNG, and what's deliberately fixed (this is a 3-act swipe roguelike,
   not a universal kit — set expectations honestly; see the non-goals in
   `docs/refactor-2-genre-neutral-core.md` §3.4).
3. **The Pack contract.** Required fields vs optional capabilities; the
   feature-detection model; the registry.
4. **The manifest.** stats/resources/paths/winGates and the HUD metadata; how a
   summit is judged (`gateValue`, partial/momentum upgrade).
5. **Authoring a deck.** `GameEvent`/`Choice`/`Outcome`; `requires` gating;
   weights, acts, path affinity, chains, flashpoints, shop/opportunity slots.
6. **Effects & the open vocabulary.** The core verb set; adding your genre's
   verbs via declaration merging; the "no unknown verb" invariant.
7. **Subsystems (plugins).** The hook lifecycle, ordering/`priority`, the
   per-resolution context, `applyResource` for owning a resource, `effectVerbs`.
   Walk the `clues` plugin (tiny) then the `songs` plugin (full).
8. **Progression: perks, interstitials, tutorial.** Pack-declared modifiers.
9. **The Presenter.** Endings, exit interviews, epilogue/headlines/DMs, trophies
   — how a genre owns its finale and flavor.
10. **Balance & the safety net.** Deterministic sims + `--check` gates, golden
    masters, cross-pack invariants, the content linter, the headless-Chromium
    UI smoke test. Teach devs to build the *same net* for their game.
11. **Shipping.** `npm run build`, the data-driven multi-entry build, save
    namespacing + pack-tagged codes, the telemetry pack dimension.
12. **Reference** (autogenerated) + a **cookbook** of recipes ("add a resource",
    "add a subsystem", "give a genre its own perks", "re-baseline a golden").

## 6. Constraints & gotchas the docs must convey (and you must respect)

- **Pinned toolchain.** `npm install` first — it pins TypeScript 5.7.3. The
  golden oracle is emit-sensitive; a stray `npx tsc` pulls a newer compiler.
- **Build before you test.** Tools/tests import the built `dist/`, not source.
- **Seeded, reproducible runs.** A run replays byte-identically from its seed;
  the goldens pin runtime behavior. If your doc examples show traces, they're
  stable. If you touch engine behavior while building tooling, expect the
  goldens to guard you (a diff is a bug unless intended).
- **The engine is genre-*neutral*, not infinitely general.** Act count, tier
  count, the swipe, the crossroads are baked in on purpose. The docs should
  frame the engine as "a 3-act swipe-roguelike SDK," not "make any game." Under-
  promising here keeps the audience happy.

## 7. Suggested first moves for the next session

1. Read the three packs (probe → mystery → music) and `js/types.ts` end to end.
   That's ~90% of the mental model in under an hour.
2. Stand up a TypeDoc pass over `js/types.ts` + `js/engine.ts` exports and see

3. Draft the **Quickstart** first, built literally on the probe pack, and get it

4. Then fill chapters 2→7 (core → subsystems), each with a real excerpt from an
   existing pack.
5. Wire a docs build into CI and (optionally) a docs entry alongside the games.

## 8. Open questions to decide early

- **SSG choice** (Docusaurus vs VitePress vs Starlight) and **how autogenerated
  reference is embedded** (TypeDoc-as-markdown into the SSG, vs a separate API
  section). Pick based on how tightly you want prose and reference interlinked.
- **Where the site lives** (a `docs-site/` in this repo built to Pages
  alongside the games, vs a separate deploy). The current Pages setup deploys
  `dist/`; a docs entry would need a build/route decision.
- **Runnable examples in the browser?** A "try it" playground (embed the engine
  + a tiny pack in a sandbox) would be SOTA but is a stretch goal — note it,
  don't block on it.
- **Versioning** — the engine has no released version yet (`package.json` is
  `0.0.0`). Decide whether docs track `main` or a tagged release before writing
  "install" instructions.
